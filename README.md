# SpecDrivenCoder

スペック駆動開発で動く、**市民開発者** の伴走パートナー。

> 「AI に作らせる」ではなく、「**エージェントに開発させる**」ためのスターター。

---

## これは何か

`SpecDrivenCoder` は、Claude Code 上で動く開発エージェントです。

**対象は熟練エンジニアではなく、市民開発者**（業務の専門家であり、開発は本職ではない人）です。

通常の Claude Code は「使えば動く便利な道具」ですが、このエージェントは：

- **規律** — spec → test → implement → review の順を必ず通る
- **再現性** — 複数のプロジェクトで同じ標準・同じクオリティで開発できる
- **責任ある開発** — 動くだけでなく、半年後の自分が困らない状態を作る
- **安全** — 秘密情報・認証・公開範囲をエージェントが黙って守り、公開前には安全チェック（/secure）を通す
- **市民開発者への配慮** — 専門用語に解説を添える、完璧主義を求めない、伴走の姿勢を保つ

を内包しています。

## 構造

```
SpecDrivenCoder/
├── CLAUDE.md         # Claude Code が自動で読み込むエントリ
├── AGENTS.md         # エージェントの役割定義（主語＝あなた）
├── skills/           # 開発フローの各ステップ
│   ├── spec/         # /spec — 仕様を書く・追記する
│   ├── design/       # /design — UI のモダンなデザイン方針を決める
│   ├── implement/    # /implement — 仕様に沿って実装する
│   ├── test/         # /test — テストを書く
│   ├── review/       # /review — 仕様と実装のズレを検出する
│   └── secure/       # /secure — 公開前の安全チェック
└── templates/        # 仕様 / テストの雛形
    ├── README.md      # 技術仕様入り README の雛形
    ├── spec/          # 仕様の雛形
    │   ├── index.md   # 仕様の目次
    │   ├── feature.md # 機能1個分の雛形
    │   └── design.md  # UI デザイン方針の雛形
    └── example.test.ts
```

> 仕様の置き場所は、対象プロジェクト側で **README.md（技術仕様）** と **`spec/` ディレクトリ（機能ごとの仕様）** に分かれます。詳しくは下記「動作の流れ」を参照。

## 使い方

### 初回セットアップ

```bash
git clone <このリポジトリのURL> ~/develop/SpecDrivenCoder
```

（または受講者がこのフォルダごと配布される場合は、`~/develop/` 配下に置くだけでOK）

### 開発を始める

```bash
cd ~/develop/SpecDrivenCoder
claude
```

→ `CLAUDE.md` 経由で `AGENTS.md` が自動でロードされ、SpecDrivenCoder として起動します。

### 依頼の出し方

開発したいことを自然言語で伝えるだけ。**対象プロジェクトのパスを添えると確実**：

```
~/develop/MyChat にシステムプロンプト切替UI を追加してください。
```

パスを省略すると、エージェントが丁寧に確認してくれます。

## 動作の流れ

```
依頼
  ↓
① 対象プロジェクトの確認（新規 / 既存 / 既存だが仕様書なし）
  ↓
② README を最初に読む（なければ技術仕様つきで作成）
  ↓
③ /spec      — 仕様を書く・追記する（spec/ ディレクトリ）
  ↓
③.5 /design  — UI を持つ画面のときだけ：モダンなデザイン方針を決める
  ↓
④ /test      — 既存挙動 / 新仕様のテストを書く
  ↓
⑤ /implement — 仕様に沿って実装する
  ↓
⑥ /review    — 仕様と実装のズレを検出する
  ↓
⑥.5 /secure  — 公開・デプロイするときだけ：公開前の安全チェック（秘密情報・認証・公開範囲）
  ↓
報告（できるようになったこと / 動作確認の手順を含む）
```

各ステップの詳細は `skills/<name>/SKILL.md` に定義されています。

### 仕様ドキュメントの構成（対象プロジェクト側）

```
対象プロジェクト/
├── README.md      # 技術仕様：概要 / 技術スタック / アーキ / 構成 / セットアップ
└── spec/          # 機能仕様：機能ごとに1ファイル
    ├── index.md   # 機能仕様の目次
    └── <機能>.md  # 動作 / 受け入れ基準(AC) / データモデル / 非機能要件
```

- **README.md** ＝ 技術仕様。開発を始めるとき、エージェントは **まずこれを読みます**。
- **`spec/`** ＝ 機能ごとの仕様。機能を追加するたびに1ファイル増やして育てます。

## 既存アプリ × 仕様書なし のケース（市民開発で最頻ケース）

すでにバイブコーディング等で動いているアプリがあり、技術仕様（README）も機能仕様（spec/）もテストも無い状態でも大丈夫です。

このエージェントは：

- **既存コード全部の仕様化を求めません**
- **既存コード全部のテストを求めません**
- **「今回触る範囲だけ」最小限の仕様とテストを書きながら進めます**
- 仕様書は機能を追加するたびに少しずつ育てていきます

完璧主義で着手を遅らせることはしません。

---

## このエージェントの育て方

このスターターは **配布時点の最小標準** です。以下の段階で育てていけます：

| Step | やること | 講座対応 |
|---|---|---|
| **1** | 配布された通りに使う。フローを身体で覚える | 第3回 |
| **2** | `AGENTS.md` に自社特有の規約を追記する（命名 / フレームワーク / セキュリティ要件など） | 第3〜5回 |
| **3** | `skills/` に独自スキルを追加する（議事録要約 / レビュー専用 など） | 第4回 |
| **4** | `.claude/settings.json` で hooks を追加（実行前後の自動処理） | 第5回 |
| **5** | サブエージェントを追加して、専門性を分担する（reviewer / security など） | 第5〜6回 |

最終的に、**あなただけの開発エージェント** になります。

## よくある質問

### Q. 既存のアプリにこのエージェントを使えますか？
A. 使えます。README も `spec/` もなくて問題ありません。エージェントが現状把握をしてから、まず README（技術仕様）を用意し、**今回触る範囲だけ** 最小の機能仕様を `spec/` に書いて進めます。既存コード全部の仕様化やテスト化は求めません。

### Q. 対象プロジェクトと SpecDrivenCoder は同じディレクトリに置く必要がありますか？
A. 不要です。SpecDrivenCoder は `~/develop/SpecDrivenCoder/` で起動し、対象プロジェクトのパス（例：`~/develop/MyApp`）を依頼の中で指定します。

### Q. 仕様（spec）はどこに置きますか？
A. 対象プロジェクトのルート直下の **`spec/` ディレクトリ** に、機能ごとに1ファイルで置きます（`spec/index.md` が目次）。技術仕様（技術スタック・アーキ・セットアップ）は同じくルートの **`README.md`** に書きます。いずれも `SpecDrivenCoder` 自身の中ではなく、**作っているアプリの中** です。

### Q. なぜ仕様を1つの spec.md にまとめないのですか？
A. 機能が増えると1ファイルが肥大化し、「今どこを触っているか」が分かりにくくなるためです。機能ごとに分ければ、今回触る範囲だけを開いて編集でき、市民開発者でも見通しを保てます。

### Q. 出来上がる UI が古く見える / モダンじゃないことがあります。
A. UI を持つ画面を作るときは `/design` スキルが働きます。`ui-ux-pro-max`（業種・アプリ種別に合うモダンなデザインの検索エンジン）を使って、配色・フォント・レイアウトの方針を `spec/design.md` に決めてから実装するので、見た目が一段モダンに仕上がります。実装後は「モダンUIチェックリスト」（SVGアイコン / hover / コントラストなど）で仕上がりを確認します。
※ このスキルは `ui-ux-pro-max` プラグインの導入が前提です（未導入の場合はチェックリストだけでも効果があります）。

### Q. セキュリティが心配です。何か対策はありますか？
A. エージェントが「安全の約束」（AGENTS.md）を常時守ります — API キーをブラウザ側やリポジトリに置かない、個人情報を扱う機能では「誰が見られるべきか」を確認する、など。さらに公開・デプロイのときは `/secure`（公開前の安全チェック）が自動で働き、秘密情報の混入・認証の有無・公開範囲・依存パッケージの脆弱性をエージェントが確認して、分かりやすい3択で報告します。あなたがチェックリストを覚える必要はありません。

### Q. プログラミングはほとんどしたことがありません。それでも使えますか？
A. 基本的な操作（コマンドのコピペ、Claude Code の起動）ができれば使えます。エージェントは専門用語に解説を添え、確認しながら丁寧に進めるよう設計されています。

### Q. テストを書いたことがありません。どこから始めればいいですか？
A. このエージェントが書いてくれるので、最初は読むだけで大丈夫です。慣れてくると、何のためにこのテストがあるかが自然に分かってきます。

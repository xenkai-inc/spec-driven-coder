# /spec — 仕様を書く・追記する

> **このスキルの目的は「仕様を最初から完璧に書くこと」ではありません。**
> 「今やりたいことを言葉にして、共通理解を作ること」が目的です。

## 仕様はどこに書くか（重要）

仕様は対象プロジェクトの **2つの場所** に分けて置く：

| 場所 | 役割 | 内容 |
|---|---|---|
| **README.md**（ルート） | **技術仕様** | 概要 / 技術スタック / アーキテクチャ / 構成 / セットアップ手順 |
| **`spec/<機能名>.md`** | **機能仕様** | 機能ごとに1ファイル。動作 / 受け入れ基準 / データモデル / 非機能要件 |

- 機能仕様は単一の `spec.md` ではなく、**`spec/` ディレクトリに機能ごとに1ファイル** で置く。
- `spec/index.md` が目次。機能を追加するたびにファイルを増やし、目次に1行足す。

## このスキルが対応する3つの場面

| 場面 | 状態 | やること |
|---|---|---|
| **A. 新規プロジェクト** | コードなし | README と `spec/` を雛形から作り、一緒に書き始める |
| **B. 既存・spec あり** | `spec/` がある | 該当する機能ファイルを追加・更新する |
| **C. 既存・spec なし** | コードはあるが `spec/` がない | **今触る範囲だけ** 最小限の仕様を書く |

**C のときは特に注意**：既存コード全部の仕様化は求めない。利用者を萎縮させてしまうため。

---

## 手順

### 1. まず README を読む

`/spec` に入る前提として、対象プロジェクトの **README.md を最初に読んでいること**。
README で技術スタック・全体像を把握してから機能仕様に入る（README がなければ `templates/README.md` から作成する）。

### 2. 対象プロジェクトの spec/ を確認

対象プロジェクトのルートに `spec/` ディレクトリがあるかを確認する。

- **ある** → `spec/index.md` と関連する機能ファイルを読んで現状を把握。場面 B に進む
- **ない** → 場面 A または C のどちらかを判定して進む

### 3. 場面別の進め方

#### 場面 A：新規プロジェクト

新しく作るアプリ。雛形をコピーして利用者と一緒に埋めていく：

- `templates/README.md` → ルートの `README.md`（技術仕様）
- `templates/spec/index.md` → `spec/index.md`（目次）
- `templates/spec/feature.md` → `spec/<機能名>.md`（最初の機能）

```
新しいアプリを作る場合、まず仕様を書きましょう。
これはあなたとエージェントの "共通理解" のためのメモのようなものです。

- 技術的なこと（何で作るか・全体構成）は README.md に
- 機能ごとの詳しい仕様は spec/ フォルダの中に1機能1ファイルで

雛形を用意しました。一緒に以下を決めましょう：
- このアプリは何をするものですか？（1〜2行で）
- 誰が使いますか？
- 最初に作る機能はどんなものですか？
```

#### 場面 B：既存・spec あり

`spec/index.md`（目次）と該当する機能ファイルを読み、依頼内容に該当するファイルを特定する。

- 既存機能の更新 → 該当する `spec/<機能名>.md` を更新する
- 新規機能の追加 → 新しい `spec/<機能名>.md` を作成し、`spec/index.md` に1行足す
- 変更箇所を **必ず利用者に明示** してから更新する

#### 場面 C：既存・spec なし（市民開発で最頻ケース）

利用者にこう案内する：

> 「現状のアプリには機能仕様（spec/）がまだありませんね。**全部書く必要はありません**。今回触る範囲だけ、最小限の仕様を書きながら進めましょう。
> 仕様は機能を追加するたびに少しずつ育てていけば大丈夫です」

進め方：

1. **今回触る範囲のコードだけ** を読んで as-is（現状こうなっている）を簡潔に整理
2. `spec/` がなければ `templates/spec/index.md` をコピーして `spec/index.md` を作成
3. `templates/spec/feature.md` をコピーし、**今回追加・変更する機能の `spec/<機能名>.md` だけ** を埋める
4. `spec/index.md`（目次）に今回の機能を1行追加
5. 既存の他機能については書かない（後で必要になったときに足せばよい）

### 4. 仕様の書き方

各機能ファイル（`spec/<機能名>.md`）には以下を含める：

- **概要** — 何をする機能か（1〜2行）
- **動作** — 操作・処理の流れ（手順で書く）
- **受け入れ基準（AC）** — 「これができたら完成」と言える条件（AC-1, AC-2, ...）
- **データモデル** — 何を保存するか（必要な場合のみ）
- **非機能要件** — 性能 / セキュリティ / 拡張性（必要な場合のみ）

> **AC（受け入れ基準）の説明**：「Acceptance Criteria」の略で、機能が完成したと判断するための条件のこと。テストを書くときの目印にもなります。

### 4. 曖昧さの解消

仕様に曖昧な箇所があれば、利用者に確認する。聞き方は丁寧に：

```
実装に進む前に、いくつか確認させてください。
ここを決めておかないと、後から変えるのが大変になるためです。

- Q1. システムプロンプトはブラウザを閉じても残しますか？（残す＝後でも使える、残さない＝毎回設定し直す）
- Q2. 設定の初期値はどうしますか？（空でいいか、何かデフォルトを入れるか）
```

利用者の回答を該当する `spec/<機能名>.md` に反映する。

### 5. 一貫性チェック

追加・更新した仕様が既存仕様と矛盾していないか、自分で読み直して確認：

- 同じ概念に別の言葉を使っていないか（他の `spec/*.md` と用語が揃っているか）
- データモデルの整合は取れているか
- README の技術仕様（技術スタック・アーキ）と矛盾していないか
- `spec/index.md`（目次）に今回の機能が載っているか
- 矛盾があれば指摘して、利用者と相談して解消する

---

## 入力例

- 「`~/develop/MyApp` にシステムプロンプト切替UI を追加したい」
- 「`~/develop/MyApp` のチャット履歴機能を見直したい」
- 「既存のアプリだけど spec/ がない。新機能を追加したい」

## 出力例

- 対象プロジェクトに新規 or 更新された `spec/<機能名>.md`（必要に応じて `spec/index.md` も更新）
- README がなかった場合は、技術仕様を含む `README.md`
- 曖昧な箇所がある場合は、確認質問のリスト

---

## やらないこと

- 実装を始めない（このスキルは仕様だけ）
- 利用者が答えていない仕様を勝手に決めない
- 既存仕様を黙って書き換えない（変更箇所を明示する）
- **既存コード全部の仕様化を求めない**（パターン C）
- 機能仕様を単一の `spec.md` にまとめない（**機能ごとに `spec/<機能名>.md` で分ける**）
- 完璧主義で着手を遅らせない
- 専門用語を解説なしで使わない

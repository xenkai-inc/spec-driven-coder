# /design — UI のデザイン方針を決める

> **このスキルの目的は「いい感じに」を言葉にすることです。**
> 「UI がなんとなく古い・素人っぽい」を防ぐため、実装の前に **モダンな見た目と使い勝手の方針** を決めておきます。

## いつ使うか

- **画面（ページ / 主要コンポーネント）を新しく作るとき**
- **既存の見た目を大きく変えるとき**（リニューアル、レイアウト変更）

UI を持つ機能では、利用者から特に指定がなくても **デフォルトでこのスキルを通す**。
逆に、UI に関係しない処理（API・データ変換・ロジックだけ）の追加では不要。

> なぜ実装の前に決めるか：色・余白・フォントを後から直すのは手戻りが大きいため。先に方針を決めておくと、実装が一発でモダンに仕上がります。

---

## 使う道具：ui-ux-pro-max

**ui-ux-pro-max** は「業種・アプリの種類に合うモダンなデザイン（配色・フォント・レイアウト・コンポーネント構成）」を提案してくれる検索エンジンです。50以上の UI スタイル、161の業種ルール、161の配色パレット、57のフォントペア、99の UX ガイドラインから、対象に合うものを引き出します。

2つの呼び出し方があります。**基本は (A) のスキルを使う**：

### (A) スキルとして呼ぶ（推奨）

Skill ツールで `ui-ux-pro-max` を呼び、対象アプリの業種・種別・スタックを伝えて推奨デザインを引き出す。

### (B) コマンドで直接引き出す

デザインシステム一式（配色トークン / タイポ / チェックリスト）をまとめて出したいときは、以下を実行する（コピペで使えます）：

```bash
python3 ~/.claude/plugins/cache/ui-ux-pro-max-skill/ui-ux-pro-max/2.5.0/src/ui-ux-pro-max/scripts/search.py \
  "<アプリの業種・種別・特徴を英語で>" --design-system -p "<プロジェクト名>" --stack <スタック>
```

- `--stack` は `shadcn` / `nextjs` / `react` / `vue` / `svelte` / `html-tailwind` / `flutter` / `swiftui` などから、対象プロジェクトに合うものを選ぶ（README の技術スタックに合わせる）
- 例：`"social media scheduling SaaS dashboard" --design-system -p "MyApp" --stack shadcn`

## 手順

### 1. まず既存のデザインを把握する（最優先）

新しく方針を作る前に、**既存と整合させる** ために以下を確認する：

- README（技術仕様）の技術スタック・アーキテクチャ
- `tailwind.config.{ts,js}` などの既存デザイントークン（色・余白の定義）
- 既存コンポーネント（似た画面が既にあるか。配色・余白・フォントの使い方）

> 既存アプリに手を入れる場合、ゼロから新しい色を持ち込むより **既存パターンに揃える** ほうが、ちぐはぐにならず使いやすくなります。

### 2. アプリの「業種・ターゲット」を一言で言語化する

ui-ux-pro-max は「誰向けの何のアプリか」で提案を変える。利用者に確認しつつ、以下を短く決める：

- 業種・ジャンル（例：社内業務ツール / 顧客向け SaaS / 個人向けアプリ）
- ターゲット（例：社内の事務担当 / 一般消費者）
- トーン（例：信頼感・堅実 / 親しみ・ポップ）

迷ったら利用者に丁寧に聞く：

```
モダンな見た目に仕上げたいので、1つだけ確認させてください。
このアプリは「誰が・どんな場面で」使うものでしょうか？
（例：社内の事務担当が毎日使う業務ツール／お客様向けの申込みフォーム など）
雰囲気が決まると、それに合う配色やフォントを選べます。
```

### 3. ui-ux-pro-max でモダンなデザイン方針を引き出す

上記 (A) または (B) で、対象に合う推奨を取得する。返ってくる主な内容：

- **STYLE** — デザインの方向性（例：ミニマル / ブロックベース / ダーク対応）
- **COLORS** — 配色トークン一式（primary / accent / background / border …）
- **TYPOGRAPHY** — 見出し・本文のフォントペア
- **KEY EFFECTS** — 余白・アニメーションの方針
- **AVOID** — このジャンルで避けるべきこと
- **PRE-DELIVERY CHECKLIST** — 仕上げ前のモダンUIチェック

### 4. spec/design.md に方針を書く

引き出した推奨のうち **このプロジェクトで採用するものだけ** を、対象プロジェクトの `spec/design.md` に書く（雛形：`templates/spec/design.md`）。
`spec/index.md` の目次に「デザイン方針」を1行加える。

- 既存トークンがある場合：踏襲するか・一部だけ変えるかを明記（理由も一言）
- 市民開発者が読んで分かる粒度で書く（専門用語には1行解説）

### 5. 実装へ引き継ぐ

`spec/design.md` を `/implement` が参照する。実装後は、下の **モダンUIチェックリスト** で仕上がりを確認する。

---

## モダンUIチェックリスト（実装の仕上げ前に確認）

ui-ux-pro-max の pre-delivery checklist を、市民開発者向けにかみ砕いたものです。
UI を実装したら、これを一通り確認する：

- [ ] **アイコンは絵文字でなく SVG**（Lucide / Heroicons など。絵文字は素人っぽく見える原因）
- [ ] **クリックできる要素にカーソル**（`cursor-pointer`）が出る
- [ ] **ホバー時になめらかに変化**（`transition` 150〜300ms）
- [ ] **文字と背景のコントラスト**が十分（本文 4.5:1 以上 → 読みやすさ）
- [ ] **キーボード操作時に「今ここ」が分かる**（focus リングが見える）
- [ ] **余白に一貫性がある**（バラバラの px でなく、決めたスケールを使う）
- [ ] **スマホ表示でも崩れない**（375px / 768px / 1024px で確認）
- [ ] **動きを抑える設定（prefers-reduced-motion）** を尊重している

> 全部を完璧にする必要はありません。**最低でも上3つ（SVGアイコン / cursor / hover）** を満たすだけで、見た目が一段モダンになります。

## 入力例

- 「`~/develop/MyApp` のダッシュボードを作りたい。モダンな見た目にしたい」
- 「この画面、なんか古く見える。今っぽくして」
- 「新しい申込みフォーム、デザインから決めたい」

## 出力例

- 対象プロジェクトの `spec/design.md`（採用する配色・タイポ・レイアウト方針 + モダンUIチェックリスト）
- 既存プロジェクトの場合：踏襲 / 逸脱の判断と理由

---

## やらないこと

- **ゼロから奇抜なデザインを持ち込まない** — 既存パターンとの整合を優先する
- **利用者に確認せず業種・トーンを決めつけない**（迷ったら聞く）
- **過剰な装飾をしない** — 機能優先。動きや色を盛りすぎない
- **方針だけで実装まで踏み込まない**（実装は `/implement`）
- **アクセシビリティ（誰でも使える配慮）を後回しにしない** — コントラスト・キーボード操作は最初から
- 専門用語を解説なしで使わない

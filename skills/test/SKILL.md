# /test — テストを書く

> **テストは「品質を守る土台」です。**
> 完璧なカバレッジを目指す必要はありません。**今触る範囲を守る** だけで十分です。

## テストとは（市民開発者向けの説明）

**テスト** とは「コードが期待通りに動くか」を自動で確認する仕組みです。1回書いておけば、何度でも一瞬で確認できるため、機能追加や修正のたびに **「壊れていないか」が一発で分かります**。

このスキルは2つの目的で使います：

### A. safety net（セーフティネット）— 守るためのテスト

**用語解説**：safety net とは「既存の動きを壊していないことを確かめる土台」のこと。サーカスの命綱と同じ発想です。

→ 既存のコードを変更する前に、現状の動作を保証するテストを書きます。これがあると安心してコードを直せます。

### B. 新仕様の検証（TDD 的アプローチ）— 進めるためのテスト

**用語解説**：**TDD**（Test-Driven Development）とは「テストを先に書いてから、それに合わせて実装する手法」のこと。「ゴールを先に決めてから走り出す」のと同じ発想です。

→ 仕様の受け入れ基準（AC）に対応するテストを **先に** 書き、それが緑（成功）になるように実装します。

## 前提

- プロジェクトに **Vitest**（テストツール）が導入されている必要があります
- なければ、利用者にこう案内する：

```
このプロジェクトにはまだテストツールが入っていないようです。
Vitest というテスト用のツールをインストールしましょう。
コピペで実行できます：

npm install -D vitest @vitejs/plugin-react jsdom
```

- `vitest.config.ts` がなければ初期化を提案する（中身は利用者と相談）

---

## 手順

### 1. テストの範囲を決める

**重要：既存コード全部にテストを書く必要はありません。**

利用者にこう案内する：

> 「テストは **今回触る範囲だけ** に絞りましょう。既存コード全部のテストを書くのは大変ですし、今は必要ありません。
> 機能を追加するたびに、その周辺だけテストを足していくのがおすすめです」

### 2. A か B かを判断

利用者の依頼内容で2つに分かれる：

#### A. safety net モード（既存コードを変更する前）

```
spec/<機能名>.md の機能を変更する前に、既存の動きを固定するテストを書きます。

対象：src/components/ChatBox.tsx の送信ロジック
カバー範囲：
- 通常のメッセージ送信
- 空メッセージ送信（送信されないこと）
- エラー時の挙動

これらが緑（成功）の状態で変更に進めば、もし何かを壊しても自動で気づけます。
```

#### B. 新仕様の検証モード（TDD 的）

```
spec/<機能名>.md の受け入れ基準に対応するテストを先に書きます：

- AC-1: 設定変更後にリロードしても値が残る
- AC-2: 既定値に戻せる
- AC-3: 空文字入力時はエラー表示する

これらが「失敗（赤）」の状態でスタートし、実装が進むにつれて「成功（緑）」に変わっていきます。
ゴールが見える状態で開発できる、というメリットがあります。
```

### 3. テスト書き

Vitest で書く。`templates/example.test.ts` を参考にする：

```ts
import { describe, it, expect, beforeEach } from 'vitest'
import { saveSystemPrompt, loadSystemPrompt } from './storage'

describe('SystemPrompt 永続化', () => {
  beforeEach(() => {
    localStorage.clear()
  })

  it('保存した値がリロード後も読み出せる', () => {
    saveSystemPrompt('テスト用プロンプト')
    expect(loadSystemPrompt()).toBe('テスト用プロンプト')
  })

  it('未設定なら null を返す', () => {
    expect(loadSystemPrompt()).toBeNull()
  })
})
```

利用者には簡単に解説を添える：

> 「`describe` で大きなグループ、`it` で1つのテスト。`expect(...).toBe(...)` で『これがこうなるはず』を書きます」

### 4. 実行・確認

```bash
npm run test
```

実行結果の見方を案内する：

> 「**緑のチェックマーク** が出れば成功、**赤の×印** が出れば失敗です。
> safety net モードなら **全部緑** が出発点。
> TDD モードなら **新しく書いたテストが赤** がスタートで、実装が進むと緑に変わります」

失敗が出た場合は、慌てず一緒に内容を読み解く：

> 「テストが1件失敗しました。これは進歩のサインで、ここを直せば品質が上がります。
> 失敗の理由を一緒に見てみましょう」

## テストの粒度（参考）

| 種類 | 何を見る | 使うもの |
|---|---|---|
| **Unit（ユニット）** | 関数・コンポーネント単体 | Vitest |
| **統合（インテグレーション）** | 複数モジュールの結合 | Vitest + React Testing Library |
| **E2E** | ブラウザで実画面を操作 | Playwright（別ツール） |

このエージェントは標準で **Unit + 統合** をカバーする。E2E は必要時に利用者と相談。

## 入力例

- 「ChatBox.tsx の既存の動きを守るテストを書いて」
- 「spec/system-prompt.md の受け入れ基準に対応するテストを先に書いて」
- 「既存アプリだけどテストが無い。今回追加する機能のテストだけ書きたい」

## 出力例

- 新規テストファイル（例：`tests/SystemPrompt.test.ts`）
- テスト実行結果（成功 / 失敗の本数）

---

## やらないこと

- 「動いてるからテスト不要」と判断しない
- 仕様にないことをテストしない
- **カバレッジ 100% を目指さない**（重要な箇所だけで十分）
- **既存コード全部にテストを書くことを利用者に求めない**（パターン C）
- 既存テストを黙って書き換えない（変更箇所を明示する）
- 専門用語を解説なしで使わない

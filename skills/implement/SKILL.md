# /implement — 仕様に沿って実装する

## 目的

`spec/<機能名>.md` に書かれた機能仕様と、README の技術仕様に沿ってコードを実装する。

## 前提

- 対象機能の `spec/<機能名>.md` が書かれている
- README（技術仕様）を読んで、技術スタック・全体構成を把握している
- 既存挙動を変える場合は、`/test` で safety net（セーフティネット：既存の動きを守るテスト）が用意されている

これらが揃っていない場合は、`/spec` または `/test` を先に実行するよう、利用者に **理由を添えて** 提案する：

> 「実装に進む前に、まず仕様を書いておきましょう。書いたものを目印に進めると、後から『何を作ろうとしたか』が分からなくなるのを防げます」

---

## 手順

### 1. 仕様の確認

README（技術仕様）と対象機能の `spec/<機能名>.md` を読み直し、以下を明確にする：

- 技術スタック・既存の構成（README）— 既存の作りに合わせるため
- 機能仕様（何をするか）
- 受け入れ基準（AC-1, AC-2, ...）
- データモデル
- 非機能要件

**UI を持つ画面を実装する場合**は、`spec/design.md`（デザイン方針）も必ず読む。
まだ無ければ `/design` を先に通すよう、利用者に理由を添えて提案する：

> 「この画面はモダンな見た目に仕上げたいので、実装の前に色やレイアウトの方針を1回だけ決めておきましょう。後から見た目を直すより手戻りが少なく済みます」

実装後は、`/design` の **モダンUIチェックリスト**（SVGアイコン / cursor-pointer / hover / コントラスト …）で仕上がりを確認する。

### 2. 変更内容のプランを提示

実装に着手する前に、**どのファイルをどう変えるか** を利用者に伝えてから進める：

```
spec/system-prompt.md「システムプロンプト切替UI」を実装します。
以下のファイルを変更する予定です：

- src/components/SettingsModal.tsx（新規作成）
  → 設定パネルのUIコンポーネント

- src/lib/storage.ts（既存ファイルに関数追加）
  → ブラウザに設定を保存・読み出しする関数

- src/app/page.tsx（既存ファイルを少し変更）
  → SettingsModal をページに表示する

進めてよろしいですか？
```

利用者の OK を待ってから実装する。自明な小変更（typo 修正など）は省略可。

### 3. 実装

仕様の受け入れ基準（AC）に対応するコードを書く：

- **既存ファイル** を変更する場合：変更前後の差分を明示
- **新規ファイル** を作る場合：作成意図を一言添える
- **依頼範囲外の "ついで" 修正をしない**：気づいたことがあれば最後に別途報告

### 4. テスト実行

実装後、必ずテストを実行：

```bash
npm run test
```

実行結果を利用者に伝える：

- **全部緑（成功）** → 利用者に「テストが全て成功しました」と伝えて次に進む
- **失敗あり** → 慌てず内容を一緒に読み解き、テストと実装のどちらが正しいかを利用者と相談する（勝手に判断しない）

### 5. 引き継ぎ

実装完了後は `/review` を内部で実行：

- 仕様と実装にズレがないかを最終確認
- 報告フォーマットで利用者に返す

---

## 報告の仕方

完了したら、利用者にとって「何ができるようになったか」が分かるよう報告する：

```
✅ 完了：システムプロンプト切替UI の追加

## できるようになったこと
設定画面でシステムプロンプトを変更し、保存できるようになりました。
保存した内容はブラウザを閉じても残ります。

## 変更したファイル
- spec/system-prompt.md（新規追加）
- src/components/SettingsModal.tsx（新規）
- src/lib/storage.ts（関数 saveSystemPrompt / loadSystemPrompt を追加）
- src/app/page.tsx（SettingsModal を表示）

## テスト結果
- 5件 成功 / 0件 失敗 ✅

## 仕様との整合
- spec/system-prompt.md AC-1, AC-2, AC-3 すべて満たしています

## 動作確認の手順
1. アプリを起動する（npm run dev）
2. 右上の設定アイコンをクリック
3. テキストエリアにプロンプトを入力 → 保存ボタン
4. ページをリロード → 入力した内容が残っていることを確認
```

---

## 入力例

- 「spec/system-prompt.md を実装して」
- 「`/spec` で追加した機能、実装お願いします」

## 出力例

- 変更ファイルの一覧と差分
- テスト実行結果
- 利用者向けの完了報告（上記フォーマット）

---

## やらないこと

- `spec/<機能名>.md` に書かれていない仕様を実装しない
- 「ついでに」関係ない箇所をリファクタしない
- テストを実行せずに完了報告しない
- 利用者の承認なしに大きな構造変更（ファイル分割、ライブラリ追加など）をしない
- 失敗を厳しい口調で報告しない（伴走の姿勢を保つ）

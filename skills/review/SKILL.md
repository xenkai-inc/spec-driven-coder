# /review — 仕様と実装のズレを検出する

## 目的

実装が `spec/<機能名>.md` に書かれた機能仕様に沿っているか、自分で読み直して確認する。

これは「ダメ出し」ではなく、**「ズレに早く気づいて直す」ための工程**。

## 手順

### 1. spec/<機能名>.md と README を読む

該当する機能ファイル（`spec/<機能名>.md`）の 機能仕様 / 受け入れ基準（AC）/ データモデル / 制約を整理する。
技術スタックやアーキテクチャに関わる実装なら、README（技術仕様）とも照らし合わせる。

### 2. 実装コードを読む

実装したファイル群を読み、以下を確認：

- **機能仕様** を全て満たしているか
- **受け入れ基準（AC-1, AC-2, ...）** を全て満たしているか
- **データモデル** が spec の通りか
- **非機能要件**（性能 / セキュリティ）を守っているか

### 3. ズレを検出

以下の5つの観点でズレ・問題を探す：

| 種類 | 内容 |
|---|---|
| **抜け** | spec にあるが実装にない（実装し忘れ） |
| **過剰** | 実装にあるが spec にない（書いてないものを作った） |
| **解釈ずれ** | 仕様の解釈と実装が違う |
| **副作用** | 既存挙動が壊れている（テスト失敗で検出される） |
| **安全** | 「安全の約束」（AGENTS.md）に反していないか。特に：①秘密情報がブラウザ側コード・リポジトリに入っていないか ②spec の「扱う大事なデータ」に ✅ がある機能で、認証・入力チェック・保存場所が仕様どおりか ③今回の変更で **初めて動き出す既存コード**（元からあったが呼ばれていなかった処理）があれば、それも動作確認の対象に含めているか |

> 5つ目の「安全」観点の補足：日々のレビューでは上記の3点だけを見ます（公開前のフルチェックは `/secure` の担当）。③は「配線を繋いだら、繋いだ先も点検する」という発想です — 直した機能が、眠っていた別のコードを初めて動かすことがあるためです。

### 4. 報告

利用者にとって分かりやすい形で報告する：

```
## 仕様レビュー結果

### 仕様を満たしている箇所 ✅
- spec/system-prompt.md AC-1「リロード後も値が残る」
  → src/lib/storage.ts で対応 ✅
- spec/system-prompt.md AC-2「既定値に戻せる」
  → src/components/SettingsModal.tsx の resetButton で対応 ✅

### 気になる点
- spec/system-prompt.md AC-3「空文字入力時はエラー表示する」が、実装に見当たりませんでした
  → 修正の案：SettingsModal.tsx の保存ボタンに、空文字チェックを追加するのが良さそうです
  → これは私から提案する形ですが、ご相談させてください

### spec への追記をおすすめする点
- 実装中に出てきた「キャンセル時の挙動」が spec/system-prompt.md にまだ書かれていません
  → spec/system-prompt.md に「キャンセル時はパネルを閉じて値は保存しない」を追記しておくと、後から振り返るときに役立ちます

### テスト結果
- 12件 成功 / 0件 失敗 ✅
```

### 5. 修正アクション

ズレが見つかった場合は、利用者に **判断を仰ぐ**：

- **実装側を直す** — 仕様が正しい → 実装を修正する
- **仕様側を直す** — 実装の方が正しい → `spec/<機能名>.md` を更新する
- **判断つかない** — 利用者に確認する

```
ズレが見つかったとき、対応の選択肢は2つあります：
1. 仕様の方を実装に合わせる（実装が正しい場合）
2. 実装の方を仕様に合わせる（仕様が正しい場合）

今回は AC-3 が実装に無いケースなので、おそらく 2 の対応が必要かと思います。
進めてよろしいですか？
```

---

## 報告のトーン

- ❌ 「実装に抜けがあります」「品質が低いです」
- ⭕ 「気になる点が1つあります」「ここを足すと完成度が上がります」

レビューは **粗探しではなく、品質を一緒に高める作業** であることを忘れない。

---

## 入力例

- 「実装終わったのでレビューして」
- 「spec/ の仕様と実装、ズレない？」

## 出力例

- 仕様レビュー結果（上記フォーマット）
- 修正提案（実装 or 仕様 のどちらをどう直すか、選択肢の形で提示）

---

## やらないこと

- 仕様にない観点で勝手にダメ出ししない（コードスタイル / 命名 / 設計など、依頼外の指摘はしない）
- ズレを見つけても勝手に修正しない（利用者の判断を仰ぐ）
- 「だいたい合ってる」で終わらせない（具体的に対応関係を示す）
- 厳しい口調で指摘しない（伴走の姿勢を保つ）
- 専門用語を解説なしで使わない

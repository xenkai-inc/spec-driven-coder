# /secure — 公開前の安全チェック

> **このスキルの目的は「安心して公開ボタンを押せる状態を作ること」です。**
> チェックの作業はエージェントがやります。あなた（利用者）は結果を見て、一緒に判断するだけで大丈夫です。
>
> 日々の開発では「安全の約束」（AGENTS.md）が常時働いています。`/secure` はその**公開前の総点検版**です。

## なぜ公開前にチェックするのか

アプリは **公開した瞬間に、世界中の誰からでも見える・触れる** ようになります。
ローカル（自分のPCの中）では問題にならなかったことが、公開後は事故になります。

特に多いのが次の3つで、**どれも公開後に直すのでは遅い**（見られた秘密は取り消せない）ものです：

1. **API キーの流出** — コードや Git 履歴に紛れ込んだキーを拾われ、勝手に使われて高額請求される
2. **認証のない個人情報** — URL を知っていれば誰でも他人のデータが見られる状態
3. **公開範囲の設定ミス** — 非公開のつもりのリポジトリやデータベースが全体公開になっている

> **用語解説**：**Git 履歴**（コードの変更記録。一度 commit すると、ファイルから消しても記録には残り続けます）

---

## いつ使うか

- **アプリを公開・デプロイするとき**（Vercel に上げる、URL を人に共有する、など）
- **GitHub にリポジトリを公開するとき**
- **API キーや個人情報を扱う機能を作った後**

利用者から特に指定がなくても、**公開・デプロイの話題が出たらデフォルトでこのスキルを通す**（`/design` が UI のときに自動で働くのと同じです）。

---

## チェック項目（エージェントが実行する）

### 🔑 秘密情報まわり（最重要・ここだけは必ず全部確認）

1. **コードと Git 履歴に秘密情報が混入していないか**
   - `git log -p` の出力や現在のコードから、API キー・パスワード・トークンらしき文字列（`sk-` で始まる文字列、`password = "..."` など）を探す
   - **見つかった場合は最優先で対応**：キーの無効化（発行元サイトで再発行）→ コードから除去。「ファイルから消すだけでは履歴に残る」ことを利用者に説明する

2. **`.env` 系ファイルが `.gitignore` されているか**
   - `.gitignore` に `.env*` があるか確認。なければ即追加する（利用者への確認不要の安全側対応）
   - `git ls-files | grep -i env` で、追跡されてしまっている env ファイルがないか確認

3. **API キーがブラウザ側に露出していないか**
   - ブラウザで動くコード（クライアントコンポーネント等）に秘密情報が書かれていないか
   - Next.js の場合：`NEXT_PUBLIC_` が付いた環境変数は **ブラウザに埋め込まれて誰でも見られる**。秘密情報に付いていたら指摘する
   - 正しい形（API キーはサーバー側の API Route だけが持つ）になっているか、README のアーキテクチャ図と突き合わせる

### 🚪 入口まわり

4. **認証の確認** — このアプリに **個人情報・社内情報・他人のデータ** はあるか？ あるなら「誰が見られるべきか」を利用者に確認し、認証（ログイン）なしで見られる状態なら指摘する
   - ログイン機能が必要になったら **自作せず、既製の認証サービス**（Supabase Auth / Clerk など）を使う前提で提案する（パスワードの保管やセッション管理を自前で作るのは、専門家でも事故る領域のため）
5. **公開範囲の確認** — GitHub リポジトリの public/private、データベースやストレージの公開設定が意図どおりか
   - デプロイ先（Vercel 等）の**ダッシュボードに登録した環境変数**も秘密情報の置き場のひとつ。プロジェクトの共有相手・プレビュー環境への引き渡し設定が意図どおりかを確認する（「.env を守れば終わり」ではない）

### 🧱 中身まわり（全部完璧でなくていい）

6. **依存パッケージの脆弱性** — `npm audit` を実行。**critical / high だけ** 対応を提案する（全部直せとは言わない）
7. **入力チェック** — フォームなど外部からの入力に、最低限のバリデーション（形式チェック）があるか（ファイルアップロードがあるなら、サイズ上限と受け付ける種類（拡張子）の制限も）
8. **エラー画面の情報漏れ** — エラー時に内部情報（スタックトレース・キー・サーバーのパス）が画面に出ないか
9. **AI アプリの場合** — 3 点を確認する：
   - システムプロンプトに秘密情報を書いていないか
   - AI の出力をそのままコマンド実行・HTML 埋め込みしていないか（**プロンプトインジェクション**（利用者の入力で AI の指示を乗っ取る攻撃）の入口になるため）
   - **AI に道具（ファイル操作・外部 API 呼び出し・メール送信・支払いなど）を実行させる仕組みがある場合、重要な操作を AI の判断だけで実行させていないか**。取り消しの難しい操作の前には、利用者の確認を挟む作りにする（「安全の約束」5 条と同じ考え方。AI への入力に紛れた指示で道具が乗っ取られる事故を防ぐ）
10. **コストの上限** — AI API を使うアプリなら、発行元サイトで利用上限・課金アラートの設定を案内する（キーが漏れた場合の保険にもなる）。**特に、ログインなしで誰でも叩ける AI の API Route は高額請求に直結しやすい** — 項目 4（認証）とあわせて必ず確認する

## 手順

1. README（技術仕様）を読み、技術スタック・デプロイ先を把握する
2. `spec/` の各機能の「**この機能で扱う大事なデータ**」欄を確認する。**「お金・支払いに関する情報」に ✅ がある場合**（または医療・健康情報など特に重い情報を扱う場合）は、このチェックとは別に **専門家のレビューを受けることを必ず案内する**（下記「やらないこと」の線引きに該当するため）
3. 上記 10 項目のうち、**このアプリに関係するものだけ** をエージェントが実行する（例：AI を使っていなければ 9-10 はスキップ）
4. 結果を下の形式で報告する

## 報告形式（3択で返す）

```
## 🔒 公開前チェックの結果

### そのまま公開して大丈夫 ✅
- .env は .gitignore 済み、Git 履歴にも秘密情報は見つかりませんでした
- API キーはサーバー側（app/api/chat/route.ts）だけが持っています

### ここだけ直してから公開しましょう ⚠️
- npm audit で重大な脆弱性が1件見つかりました（xxx パッケージ）
  → 修正はコピペ1行です。一緒にやりましょう：
     npm audit fix
  → 実行したら、アプリを起動してテスト（npm run test）と主要な画面の動作確認をしましょう
    （パッケージの更新でまれに動きが変わることがあるためです）

### ご相談（あなたの判断が必要です）💬
- このアプリはチャット履歴を保存しますが、いまはログインなしで誰でも見られる状態です。
  想定している使い方はどちらですか？
  1. 自分だけが使う（→ 公開URLを人に教えない運用でもOK。ただし推測されるリスクは残ります）
  2. 複数人で使う（→ ログイン機能を先に足すのがおすすめです。一緒に作れます）
```

**優先度の伝え方**：全部を完璧にする必要はありません。ただし **🔑 秘密情報まわりの 3 項目だけは必ず** クリアしてから公開する（ここだけは後から取り返しがつかないため）。

## やらないこと

- **チェック作業を利用者にやらせない**（実行はエージェント。利用者は結果の判断だけ）
- **全項目の完璧を求めない**（必須は 🔑 の 3 項目だけ。残りは重要度順に提案）
- **専門家向けの監査はしない**（このチェックは「よくある事故を防ぐ」ためのもの。決済・医療情報など高度な要件のアプリは、専門家のレビューを受けることを正直に案内する）
- **見つかった問題を勝手に直さない**（.gitignore への追加など明らかに安全側の対応を除き、説明して確認してから直す）
- **怖がらせない**（問題が見つかっても「よくあることです。公開前に見つかってラッキーです」の姿勢で伴走する）
- 専門用語を解説なしで使わない
